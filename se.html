Unit 1

Software Engineering (BCA-4003)

1. Definition and Basics

Software Engineering (IEEE): "The application of a systematic, disciplined, quantifiable approach to the development, operation, and maintenance of software." PYQ 2024

Father of SE: Watts S. Humphrey PYQ 2024

Software Characteristics: PYQ 2022

  • Logic-based, not physical (does not wear out).
  • Engineered/Developed, not manufactured.
  • Custom-built usually, unlike hardware components.
  • Complex and invisible.

2. Generic View of Software Engineering

Software Engineering is a Layered Technology:

  • Tools: Automated support (CASE tools).
  • Methods: Technical "how-to" for building software (Analysis, Design, Coding, Testing).
  • Process: The glue that holds layers together; defines the framework (Activities, Actions, Tasks).
  • Quality Focus: The foundation; bedrock that supports software engineering.

Process Framework Activities: Communication, Planning, Modeling, Construction, Deployment.

3. Software Paradigms (SDLC Models) PYQ 2022, 2024

Waterfall Model (Linear Sequential): Classic Lifecycle. PYQ 2024
Req → Design → Coding → Testing → Maintenance. Simple but rigid.

Incremental Model: Delivers software in small increments/versions.

RAD (Rapid Application Development): High-speed adaptation of linear model using component-based construction.

Spiral Model (Evolutionary): Risk-driven. Couples iterative nature of prototyping with systematic aspects of waterfall.
Phases: Planning, Risk Analysis, Engineering, Evaluation. Best for high-risk, large projects. PYQ 2022

Prototype Model: Building a mock-up to clarify requirements early.
Use when: Requirements are unclear. User gets a "feel" of the system before actual development.

Agile Model: Iterative, incremental, flexible. Focus on customer satisfaction and responding to change (e.g., Scrum, XP).

Unit 2

Software Engineering

Requirements Analysis

  • Requirement Engineering Process: Feasibility Study, Elicitation, Analysis, and Validation.
  • SRS (Software Requirement Specification): A document describing what the system should do.
    • Characteristics: Correctness, Unambiguous, Consistent, Verifiable.
  • Data Flow Diagrams (DFD): Used to model the flow of data.
    • Level-0 DFD (Context Diagram): Represents the entire system as a single bubble interacting with external entities.

Cost Estimation Models (COCOMO)

  • COCOMO (Constructive Cost Model): Proposed by Barry Boehm to estimate project cost and effort.
  • Basic COCOMO: Estimates effort based on program size (KLOC - Kilo Lines of Code). Formula: $Effort = a(KLOC)^b$.
  • Modes of Development:
    • Organic: Small, simple projects, experienced team.
    • Semidetached: Intermediate size and complexity.
    • Embedded: Complex, hardware constraints.

Design Concepts: Coupling & Cohesion PYQ

Goal: High Cohesion and Low Coupling.

  • Cohesion (Internal strength): Measure of how related functions are within a module.
    • Functional Cohesion (Best): Module performs one specific task.
    • Coincidental Cohesion (Worst): Unrelated tasks grouped together.
  • Coupling (Inter-dependency): Measure of dependency between two modules.
    • Data Coupling (Best): Modules share data via parameters.
    • Content Coupling (Worst): One module modifies internal data of another.

Unit 3

Software Engineering

Software Design Concepts

  • Software Design: Transforming SRS into a structure for implementation.
  • Modularity: Dividing software into distinct, addressable components.
  • Coupling: Interdependence between modules. Goal: Low Coupling.
    • Worst: Content Coupling. Best: Data Coupling.
  • Cohesion: Strength of relationship within a module. Goal: High Cohesion.
    • Best: Functional Cohesion. Worst: Coincidental Cohesion.
  • Structure Chart: Graphical representation of system modules and hierarchy.
  • Design Strategies:
    • Top-Down: Start with main component, decompose into sub-components. (Stepwise Refinement).
    • Bottom-Up: Start with lowest level components (utility functions), combine to form higher levels.

Unit 4

Software Engineering

Implementation & Coding Standards

  • Implementation: Translating design into source code.
  • Good Coding Style: Focuses on readability, maintainability, and modularity.
  • Structured Programming: Uses sequence, selection, and iteration. Avoids `GOTO`.
  • Validation vs. Verification:
    • Validation: "Are we building the right product?" (User needs).
    • Verification: "Are we building the product right?" (Specs).
  • Software Testing Strategies:
    • White Box (Glass Box): Testing internal logic/code (e.g., Basis Path). Developer does this.
      • Cyclomatic Complexity: Metric $V(G) = E - N + 2P$. Measures logical complexity.
    • Black Box: Testing functionality (Input/Output) without knowing code (e.g., Boundary Value Analysis).
    • Levels: Unit (Individual modules), Integration (Interaction), System (Complete).
  • Software Configuration Management (SCM): Managing changes (Version Control) and controlling software evolution.
  • Testing Types:
    • Alpha Testing: By internal team in developer environment.
    • Beta Testing: By real users in real environment.
    • Regression Testing: Re-testing after modification to ensure no new bugs.
  • Software Maintenance:
    • Corrective: Fixing bugs.
    • Adaptive: Moving to new platform/OS.
    • Perfective: Adding new features.
    • Preventive: Future-proofing (Refactoring).

Unit 5

Software Engineering

Software Maintenance

  • Definition: Modifying software after delivery.
  • Types of Maintenance:
    1. Corrective: Fixing bugs/errors (e.g., login failure).
    2. Adaptive: Adjusting to new environments (OS/Hardware changes).
    3. Perfective: Adding features/improving performance (User requests).
    4. Preventive: Refactoring to prevent future issues.
  • Reverse Engineering: Analyzing a system to identify components and create a higher-level representation.
  • CASE Tools (Computer-Aided Software Engineering):
    • Upper CASE: Support Planning/Analysis/Design phases (e.g., Diagram tools).
    • Lower CASE: Support Coding/Testing/Maintenance (e.g., Compilers, Debuggers).
