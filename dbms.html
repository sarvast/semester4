Unit 1 Content

Database Management System (BCA-4002)

1. Introduction to DBMS

Database: A clear, organized collection of data.
DBMS: Software to define, construct, manipulate, and share databases.

Characteristics of Database Approach: PYQ 2024

  • Self-Describing Nature: Contains data + metadata (catalog).
  • Insulation between Programs and Data: Data Abstraction.
  • Support of Multiple Views: Different users see different parts of data.
  • Data Sharing & Multi-user Transaction Processing: Concurrent access.

2. Data Models

A collection of concepts for describing data, relationships, and constraints.

  • High-Level (Conceptual): Close to user view. E.g., Entity-Relationship (ER) Model.
  • Representational (Implementation): Between high and low level. E.g., Relational Model (Tables), Hierarchical (Tree), Network (Graph). PYQ 2024
  • Low-Level (Physical): How data is stored on disk.

3. DBMS Architecture PYQ 2024

Three-Schema Architecture (ANSI/SPARC):

  1. Internal/Physical Level: Physical storage details. PYQ 2024-25
  2. Conceptual/Logical Level: Structure of whole database (Tables, Constraints). Hides physical details. PYQ 2022
  3. External/View Level: User-specific views. (e.g., Student sees only grades, Admin sees everything).

Client-Server Architectures:

  • 2-Tier: Client Application ↔ Database Server (Direct).
  • 3-Tier: Client ↔ Application Server ↔ Database Server (Scalable, Secure).

4. Data Independence PYQ 2024-25

The capacity to change the schema at one level without changing the schema at the next higher level.

  • Logical Data Independence: Modify Conceptual Schema without changing External Views/Application Programs. (e.g., Adding a new column doesn't break app code).
  • Physical Data Independence: Modify Internal Schema without changing Conceptual Schema. (e.g., Changing from HDD to SSD or changing file indexing structure).

5. DBMS Languages & Users

  • DDL (Data Definition Language): Define structure (CREATE, ALTER, DROP, TRUNCATE). MCQ Topic
  • DML (Data Manipulation Language): Manage data (SELECT, INSERT, UPDATE, DELETE).
  • DCL (Data Control Language): Control access (GRANT, REVOKE).
  • TCL (Transaction Control Language): Manage transactions (COMMIT, ROLLBACK).

Database Users: DBA (Admin), Application Programmers, Sophisticated Users (Analysts), Naive Users (End-users).

Unit 2 Content

Database Management System (DBMS)

E-R Modeling (Entity-Relationship Model)

  • Entity: A real-world object (animate or inanimate) that is easily identifiable (e.g., Student, Employee). Represented by a Rectangle.
  • Attributes: Properties that describe an entity (e.g., Name, Age). Represented by an Ellipse.
    • Simple: Atomic values (e.g., Roll No).
    • Composite: Subparts (e.g., Name $\rightarrow$ First, Last).
    • Multivalued: Multiple values (e.g., Phone). Represented by Double Ovals.
  • Relationship: An association between two or more entities. Represented by a Diamond.
  • Keys:
    • Primary Key: Uniquely identifies an entity. Underlined.
    • Weak Entity: Cannot exist without a strong owner entity. No primary key. Double Rectangle.

Normalization Important

Process of organizing data to minimize redundancy and dependency.

  • 1NF (First Normal Form): Atomic values (No multi-valued attributes).
  • 2NF (Second Normal Form): 1NF + No Partial Dependency (All non-key attributes dependent on full Primary Key).
  • 3NF (Third Normal Form): 2NF + No Transitive Dependency (Non-key attribute determining another non-key attribute).
  • BCNF (Boyce-Codd Normal Form): Stricter 3NF. For every dependency $A \to B$, $A$ must be a Super Key.

Unit 3 Content

Database Management System (DBMS)

Data Normalization

  • Definition: Organizing data to minimize redundancy and dependency. Decomposes large tables.
  • 1NF (First Normal Form): Eliminates repeating groups. Attributes must be atomic.
  • 2NF (Second Normal Form): 1NF + No Partial Dependency (Non-key attributes depend on the full primary key).
  • 3NF (Third Normal Form): 2NF + No Transitive Dependency (Non-key attributes depend only on the primary key, not other non-keys).
  • BCNF (Boyce-Codd Normal Form): Stronger 3NF. For functional dependency $X \to Y$, $X$ must be a super key.
  • Decomposition Properties:
    • Lossless Join: Natural join of decomposed tables returns original table without spurious tuples.
    • Dependency Preservation: All functional dependencies can be checked within individual tables without joining.

Unit 4 Content

Database Management System (DBMS)

Relational Model & SQL

  • Relational Model: Data in tables (Relations). Row=Tuple, Column=Attribute.
  • Integrity Constraints: Entity (PK $\neq$ Null), Referential (FK matches PK), Domain (Valid values).
  • Relational Algebra:
    • Selection ($\sigma$): Horizontal subset (Rows).
    • Projection ($\pi$): Vertical subset (Columns).
    • Difference ($-$): Rows in A but not B.
    • Cartesian Product ($\times$): All combinations of rows.
  • SQL (Structured Query Language):
    • DDL: CREATE, ALTER, DROP.
    • DML: SELECT, INSERT, UPDATE, DELETE.
    • Keywords: DISTINCT (Remove duplicates), WHERE (Filter).

Transaction Management & Concurrency

  • Transaction: logical unit of work (e.g., Transfer money).
  • ACID Properties:
    • Atomicity: All or nothing. (Commit/Rollback).
    • Consistency: DB remains consistent before/after transaction.
    • Isolation: Transactions don't interfere with each other.
    • Durability: Committed changes are permanent (survive failure).
  • Concurrency Control: Managing simultaneous access.
    • Locks: Binary, Shared/Exclusive.
    • Deadlock: Two transactions waiting for each other.

Unit 5 Content

Database Management System (DBMS)

Transaction Processing & Compulsory Control

  • ACID Properties:
    • Atomicity: All or nothing.
    • Consistency: DB remains valid.
    • Isolation: Transactions don't interfere.
    • Durability: Saved permanently.
  • Concurrency Control: Ensures that simultaneous transactions do not conflict.
    • Locking Protocols: Shared Lock (Read), Exclusive Lock (Write).
    • 2PL (Two-Phase Locking): Ensures Serializability.
      • Growing Phase: Transaction acquires locks.
      • Shrinking Phase: Transaction releases locks. Cannot acquire new locks.
    • Deadlock: A situation where Transaction A waits for B, and B waits for A. Neither proceeds.
  • Recovery Techniques:
    • Log-Based (WAL): Records every transaction in a "Log" file before applying to disk.
    • Checkpoints: Periodically saving database state to disk to minimize recovery time.
    • Shadow Paging: Maintains two page tables (Current and Shadow). Updates play on new pages; Shadow becomes Current on commit.
  • Transaction States:
    • Active: Execution starts.
    • Partially Committed: Final statement executed.
    • Committed: Successful completion.
    • Failed: Normal execution cannot proceed.
    • Aborted: Transaction rolled back.
